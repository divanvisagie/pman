Project Workflow Manual

pman is an opinionated workflow for agentic programming.

LLMs are great at generating code and iterating through implementation problems, but they struggle with context. Context is the hardest part of software development. pman flips the dynamic: you become the context manager, while your coding agent focuses on code, frameworks, and documentation. Each plays to their strength.

The key insight is that most file changes happen through the agent chat as an intermediary. Instead of editing files or running commands directly, you work through your coding agent. Because the agent is configured with the workflow rules via AGENTS.md and skills, it enforces conventions automatically: creating project notes before coding, updating the registry, following commit formats. You describe intent; the agent handles execution within the established structure.

This doesn't mean you can't edit files directly. Sketch out pseudocode in vim, tweak a config by hand, or use whatever tool fits the moment. The workflow is interactive: when you make changes outside the chat, tell your agent to look at what you did. Your agent, your editor, and any other tool are tools in the toolbox—not the entire toolbox.

Unlike throwaway planning, pman treats plans as persistent artifacts, like source files, but managed in a separate, centralized Notes directory. By documenting every change, you build a reference set for future work. Changed service A and now service B needs updating? Pull in context from A's project note. The Notes vault becomes your cross-system memory.

Agent compatibility: This workflow is designed for coding agents that can read workspace guidance files and skills. pman keeps canonical rules in AGENTS.md and provides bridge symlinks for supported CLIs such as Claude and Codex.

Quick Start

  1. Create a workspace directory (e.g., ~/src, ~/code, whatever you prefer)
  2. Start your coding agent in that directory
  3. Tell your agent: "set up https://divanv.com/pman/INSTALL.md"

Your coding agent will read the install instructions and configure your workspace automatically. The setup works in any directory—it becomes your workspace root.

Glossary

TermMeaning
ProjectTime-bound effort to achieve an outcome (feature, bugfix, refactor)
SystemWhat the project changes—may span one or more repositories

A project changes a system. A system is made of one or more repositories. Projects live in Notes/Projects/, repositories live in the workspace.

The Core Idea

Plan before you code. Every change starts in a project note, not in the codebase. You and your coding agent collaborate on the plan in writing. Only after the plan is complete do you start coding. 


flowchart LR
    subgraph Workflow
        A[Create Project Note] --> B[Plan with Agent]
        B -->|Update| N[(Project Note)]
        N --> C[Execute Code]
        C --> D[Record Outcomes]
        D -->|Update| N
    end

    D --> E{Done?}
    E -->|No| B
    E -->|Yes| F[Archive]

The workflow has five steps:

  1. Create a project note: pman new creates a project note in Notes/Projects/ with a chronological PROJ-<n> id and slug.
  2. Plan collaboratively: Work with your coding agent to develop the plan in the project note. Goals, constraints, approach: all written down. The plan lives in the note, not in chat history.
  3. Execute: Once the plan is complete, start writing code. The plan is the spec; follow it.
  4. Record outcomes: Update the project note with what worked, what changed, and any follow-up tasks.
  5. Archive: pman archive moves the project note to Notes/Archives/Projects/ and updates the registry.

Example Session

Here's how a typical session looks:


sequenceDiagram
    actor User
    participant Agent
    participant PN as Project Note
    participant Code as System

    User->>Agent: I want to add feature X
    Agent->>PN: Create project note
    User->>Agent: Let's plan this
    Agent->>PN: Write goals and approach
    User->>Agent: What about edge cases?
    Agent->>PN: Update plan
    User->>Agent: Looks good, build it
    Agent->>Code: Implement changes
    Agent->>PN: Record outcomes
    User->>Agent: Done, archive it
    Agent->>PN: Move to Archives

Notice how the agent writes to the project note throughout. It does this based on the instructions in AGENTS.md, but you may also want to explicitly tell it to write something down during your chat. The note becomes the single source of truth for why a change was made, while git records what changed.

Workspace Structure

pman expects a Notes directory following PARA (Projects, Areas, Resources, Archives): 

Notes/
  Projects/
    _registry.md
  Areas/
  Resources/
  Archives/
    Projects/

The workspace structure for your repositories is up to you—document it in your workspace README.md.

Key concepts:

  • Registry: Notes/Projects/_registry.md is the authoritative index of all projects, active and archived.
  • Deterministic slugs: Each project gets a unique slug like proj-22-my-feature. Slugs make projects easy to reference in conversation and across notes.

CLI Setup

Use the pman CLI to bootstrap the workspace in the current directory:

  1. Install the CLI: cargo install --git https://github.com/divanvisagie/pman.
  2. Initialize workspace files: pman init ..
  3. Verify setup: pman verify --path ..
  4. Create a README.md documenting your workspace layout and custom tools.

See INSTALL.md for the agent setup instructions.

Install the CLI

cargo install --git https://github.com/divanvisagie/pman

Create a new project:

pman new "My Feature" --status active

Archive a completed project:

pman archive proj-22

Configuration

pman ships two types of configuration files:

AGENTS.md

Generic workflow instructions for coding agents. Contains workflow rules, commands, and boundaries. Written as directives for agents, not documentation for humans. 

cp resources/AGENTS.md ./

README.md (user-maintained)

Your workspace-specific configuration. Document your:

  • Directory layout and organization
  • Custom tools and commands
  • Project creation workflow
  • System-specific conventions

Agents read both files, combining generic workflow with your specific setup.

Agent Skills

Skills teach coding agents about specific workflows. This repo includes one merged skill in resources/skills/:

  • para-notes: PARA note management, note I/O commands, and workspace/project boundary guidance

Canonical install path is .pman/skills/:

mkdir -p ./.pman/skills/para-notes
cp resources/skills/para-notes/SKILL.md ./.pman/skills/para-notes/SKILL.md

When supported CLIs are installed, pman creates bridge symlinks:

CLAUDE.md -> AGENTS.md
.claude/skills/para-notes -> .pman/skills/para-notes
.codex/skills/para-notes -> .pman/skills/para-notes

Upgrading

Because AGENTS.md and skills are generic, upgrading is simple:

pman update --path .

Your README.md stays untouched—no merge conflicts.

Keeping Things in Sync

The AGENTS.md file is the contract between you and your coding agent. It defines commands, tools, structure, and workflow rules. Keep it up to date.

When an agent does something you don't like, ask it to update AGENTS.md directly: "Please add to AGENTS.md not to do X again." This way, the lesson persists across sessions.

Similarly, keep your project notes current. They're the shared context that makes collaboration with coding agents effective.
